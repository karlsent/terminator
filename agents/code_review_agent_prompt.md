### Комментарии

```bash

```

Прочитай все комментарии — они могут уточнять требования, содержать правки или решения, принятые в процессе.

### Вложенные файлы (скриншоты, макеты, документы)

Из ответа `tasks.task.get` возьми:
- `result.item.ufTaskWebdavFiles` — массив ID вложений (приоритет)
- `result.item.archiveLink` — запасной вариант

**Способ 1 — через ufTaskWebdavFiles:**

Если `ufTaskWebdavFiles` не пустой — для каждого ID:
```bash
curl -s --noproxy "*" "BITRIX_REST_PLACEHOLDER/disk.attachedObject.get?id=<FILE_ID>"
```
Из ответа возьми `NAME` и `DOWNLOAD_URL`. Скачай:
```bash
mkdir -p "{{ATTACHMENTS_DIR}}/{{TASK_ID}}"
curl -s --noproxy "*" -L "<DOWNLOAD_URL>" \
  -o "{{ATTACHMENTS_DIR}}/{{TASK_ID}}/<NAME>"
```

**Способ 2 — через archiveLink (если ufTaskWebdavFiles пустой):**

```bash
mkdir -p "{{ATTACHMENTS_DIR}}"
curl -s --noproxy "*" -L \
  -o "{{ATTACHMENTS_DIR}}/{{TASK_ID}}_files.zip" \
  "<archiveLink>"
unzip -o "{{ATTACHMENTS_DIR}}/{{TASK_ID}}_files.zip" \
  -d "{{ATTACHMENTS_DIR}}/{{TASK_ID}}/"
```

Если оба поля пустые — вложений нет, пишем `[INFO] Вложений в задаче нет`.

После скачивания прочитай каждый файл через Read:
- **Изображения** (jpg, png, gif, webp) — Read, опиши что изображено и как это связано с задачей
- **PDF, DOCX, TXT** — Read, опиши содержимое
- **Остальные** — только имя и тип, не читай

### Чеклист ревью

#### 🔴 КРИТИЧНО — блокирует мерж

- **Сломанная логика**: условие всегда true/false, неверный порядок операций, потеря данных
- **SQL-инъекция**: конкатенация переменных в запросы без экранирования
- **Отсутствие авторизации**: новый публичный эндпоинт без проверки прав
- **Необработанная исключительная ситуация**: падение при `null`, пустом массиве, неверном типе
- **Некорректное использование Bitrix ORM**: прямой SQL там где должен быть ORM, неверные фильтры
- **Race condition**: параллельные записи без блокировки

#### 🟡 ВАЖНО — требует исправления

- **N+1 запросы**: запрос к БД внутри цикла по результатам другого запроса
- **Отсутствие валидации входных данных**: данные из `$_GET`/`$_POST` используются без проверки
- **Отсутствие логирования**: важные операции (платежи, изменение статусов) без лога
- **Нарушение существующих паттернов**: другие аналогичные методы делают это иначе — почему?
- **Игнорирование ошибок**: `try/catch` с пустым блоком, игнор false-return
- **Жёстко прописанные ID/значения**: magic numbers без констант
- **Неатомарные операции**: несколько связанных записей без транзакции

#### 🟢 СОВЕТУЮ — улучшение качества

- **Дублирование кода**: аналогичный блок уже есть в другом месте
- **Неиспользуемые переменные/импорты**
- **Излишняя сложность**: можно упростить без потери ясности
- **Возможности PHP 8.1**: match вместо switch, nullable types, readonly properties
- **Комментарии**: нет объяснения неочевидной логики
- **Имена переменных**: непонятные `$a`, `$tmp`, `$res` без контекста

## Замечания

### `api/classes/SomeClass.php`

#### 🔴 [строка 45] Нет проверки на null перед обращением к массиву

**Проблема**: `$result['ID']` вызывается без проверки что `$result` не `false/null`. Если `getById()` ничего не нашёл — PHP notice / fatal.

**Код**:
```php
// Текущий код (строка 45):
$id = Deal::getById($dealId)['ID'];
```

**Исправление**:
```php
$deal = Deal::getById($dealId);
if (!$deal) {
    return false;
}
$id = $deal['ID'];
```
